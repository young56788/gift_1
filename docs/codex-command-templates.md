# Codex 指令模板（本项目）

适用场景：地图/钓鱼/卡坦/晚会持续迭代，快速定位问题，稳定收口。

## 1) 开工模板（先查再改）

```text
先不要改代码。先做只读检查并输出：
1) 当前分支与未提交改动
2) 本轮目标相关文件
3) 风险清单（P0/P1/P2）
4) 预计改动方案（最小改动优先）
确认后再实施。
```

## 2) 实施模板（单轮功能/修复）

```text
Implement the plan.
要求：
1) 只改与本轮目标直接相关的文件
2) 不要改视觉资源路径和无关逻辑
3) 每次关键修改后说明“改了什么、为什么”
4) 完成后执行 npm run build 并报告结果
```

## 3) 排错模板（黑屏/场景切换）

```text
先不要改代码。按下面顺序排查并给结论：
1) scene load/unload 顺序
2) currentScene / gameReady / activeScenes 一致性
3) 首个业务报错来源文件和触发步骤
4) 最小修复点（只改 1-2 处）
确认后再修复并回归：
- 地图 -> 卡坦 -> 返回地图
- 地图 -> 钓鱼 -> 返回地图
```

## 4) Subagent 模板（并行但不冲突）

```text
用 subagent 并行做两件事：
A 只读定位根因（不改代码）
B 输出手工回归脚本（不改代码）
主代理汇总后给我“单一实施方案”。
```

```text
用 subagent 并行修复：
A 负责场景管理（createPhaserGame/Host）
B 负责业务场景（Map/Shrimp）
不要改同一文件。完成后主代理统一集成、跑 build、给风险清单、关闭全部 subagent。
```

## 5) 收口模板（每轮结束固定）

```text
统一集成、跑 build、给我风险清单，再关掉所有 subagent。
另外给出：
1) 本轮已完成
2) 还剩什么
3) 下一轮建议优先级
```

## 6) 提交模板（你确认后再执行）

```text
先展示：
1) git status -sb
2) 本次改动摘要（按模块）
3) 验证结果（build + 手工路径）
我确认后再 commit，不要直接 merge master。
```

## 7) MCP 联动模板（GitHub/Vercel/Figma）

```text
用 Vercel MCP 检查最新部署状态和报错日志，给出可执行修复建议。
如果涉及 UI 偏差，再用 Figma MCP 对比指定节点并给差异清单。
最后整理成 GitHub PR 描述草稿。
```

## 8) 安全模板（密钥保护）

```text
后续涉及 token，不要让我在聊天里粘贴明文。
只给我本机终端命令，并用 set/unset 结果校验，不回显 token 值。
```
